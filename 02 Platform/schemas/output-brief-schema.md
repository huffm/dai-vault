# schema: output brief

## purpose
defines the standard shape of a brief produced by any niche workflow. the output brief is the unit of value the platform delivers to a tenant. the schema is generic; the content is niche-specific.

## principle
the brief structure is a platform concern. what goes inside each section is a niche concern. agents write to this structure; the delivery agent reads from it to format and dispatch.

---

## brief envelope

| field | type | description |
|---|---|---|
| `brief_id` | string (uuid) | unique identifier for this brief instance |
| `tenant_id` | string (uuid) | tenant this brief was generated for |
| `niche` | string (enum) | which niche produced this brief |
| `workflow` | string | which workflow produced this brief (e.g. `pre-game-brief`, `daily-market-brief`) |
| `trigger` | string (enum) | what caused this brief to run: `scheduled`, `event-driven`, `threshold-alert` |
| `generated_at` | timestamp | ISO 8601 |
| `delivery_status` | string (enum) | `pending`, `delivered`, `failed` |
| `sections` | object[] | ordered list of brief sections — see below |

---

## section structure

each entry in `sections` follows this shape:

| field | type | description |
|---|---|---|
| `section_id` | string | machine-readable label (e.g. `signal-table`, `narrative`, `header`) |
| `label` | string | human-readable section title |
| `content_type` | string (enum) | `markdown`, `table`, `score`, `flag` |
| `content` | string or object | the rendered content for this section |

---

## standard section types

| section_id | content_type | purpose |
|---|---|---|
| `header` | markdown | brief title, subject, date, and key identifiers |
| `signal-table` | table | scored signals with flag per row |
| `aggregate-score` | score | single rolled-up score or tone label |
| `narrative` | markdown | 3-5 sentence synthesis written by summarizer agent |
| `confidence-flag` | flag | `high` / `medium` / `low` based on signal agreement |
| `upcoming-events` | markdown | relevant scheduled events in the near window |
| `alert-reason` | markdown | present only on threshold-alert briefs; explains what triggered the alert |

not every brief includes every section. each niche workflow declares which sections it produces.

---

## delivery modes
a brief envelope is dispatched by the delivery agent in one of two modes:

**scheduled dispatch** — brief is generated on a fixed schedule and delivered to the tenant's configured channel at generation time.

**alert dispatch** — brief is generated immediately when a threshold is breached. delivery is immediate regardless of schedule. `trigger` field is set to `threshold-alert` and an `alert-reason` section is included.

---

## rules
- `brief_id` is generated by the platform, not the niche workflow
- niche workflows populate `sections`; they do not define envelope fields
- `delivery_status` is updated by the delivery agent after dispatch attempt
- a failed delivery should be retried once before marking `failed` and logging
- briefs are not stored indefinitely — retention policy is a platform operations decision, not defined here
